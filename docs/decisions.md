# Журнал архитектурных решений (ADR)

Формат: контекст → решение → обоснование. Решения принимались без блокирующих
вопросов, как предписано ТЗ §1.6.

## D-001. Один Next.js вместо monorepo (NestJS + Turborepo)
ТЗ §2 *рекомендует* monorepo с отдельным NestJS API. Решение: единое Next.js
приложение; REST API — route handlers в `app/api/*`, домены изолированы в `lib/*`
(границы будущих пакетов). Обоснование: критерий готовности (§21) — работающий
продукт; два приложения+общие пакеты умножают время на инфраструктуру, не добавляя
функций. Контракты API сохранены (OpenAPI в docs), извлечение в отдельный сервис —
механическая операция.

## D-002. Фоновые задачи без Redis/BullMQ
Пересчёт рейтингов и уведомления — идемпотентные функции за интерфейсом `jobs/queue.ts`
(enqueue → сейчас in-process + периодический тик). Обоснование: один инстанс на старте;
BullMQ добавил бы Redis в обязательную инфраструктуру. Своп на BullMQ — реализация
того же интерфейса.

## D-003. Файлы: локальный диск за абстракцией вместо обязательного MinIO
`FileStorage` (put/get/delete/signedUrl) с провайдером `disk` (uploads/ вне git,
HMAC-подписанные URL с истечением) — S3/MinIO провайдер подключается через env без
изменения кода. Обоснование: меньше обязательных контейнеров; интерфейс совпадает.

## D-004. Package manager — pnpm
Установлен pnpm@11; команды из ТЗ §20 (`pnpm dev` и т.д.) работают как написано.

## D-005. Rich-text уроков — Markdown + строгая санитизация
Вместо WYSIWYG: Markdown-редактор с предпросмотром; рендер через собственный
ограниченный конвертер + экранирование (без сырого HTML от пользователя).
Обоснование: XSS-поверхность минимальна (§16), формат переносим. KaTeX — отдельным
проходом для формул `$…$`.

## D-006. Сессии: httpOnly cookie + серверная таблица, ротация refresh
Короткоживущий access (cookie, 15 мин) + refresh (cookie, 30 дней, one-time — при
каждом обновлении токен заменяется, старый инвалидируется; кража старого токена
детектится по повторному использованию → сброс всех сессий пользователя).
SameSite=Lax + проверка Origin на мутациях = CSRF-защита (§16).

## D-007. Родители/преподаватели/админы — email+пароль; ученики — телефон+OTP
ТЗ фиксирует только вход ученика (телефон+OTP) и админов/преподавателей
(email+пароль). Родителям назначен email+пароль (единый безопасный путь).
`SmsProvider`: dev-провайдер пишет код в консоль сервера и (только при
`NODE_ENV!=production`) возвращает его в ответе API для e2e; боевой провайдер —
через env `SMS_PROVIDER=...`.

## D-008. «Посещаемость» для родителя = активность
Отдельной сущности Attendance в ТЗ §14 нет. Родителю показываются дни активности
(по событиям обучения) с честной подписью «Активность», плюс прогресс, оценки,
дедлайны и комментарии преподавателей.

## D-009. Сертификаты — HTML с уникальным кодом и публичной проверкой
Печатаемая страница сертификата + `/verify/[code]`. PDF-генерация не включена
(тяжёлые зависимости), формат расширяем.

## D-010. TanStack Query / React Hook Form не используются
Серверные компоненты + точечные клиентские хуки (`useApi`) и контролируемые формы
с zod-валидацией на сервере. Обоснование: меньше зависимостей — меньше поверхностей
отказа; вся валидация всё равно обязана жить на API (§16).

## D-011. Тестовая БД
Integration-тесты гоняются на отдельной схеме того же Postgres
(`DATABASE_URL_TEST`), Playwright — на dev-сервере с seed-данными.

## D-012. Организации (multi-tenant-ready)
Все учебные сущности имеют `organizationId`; seed создаёт организацию по умолчанию;
UI переключения организаций не строится (не требуется §21), но данные и RBAC готовы.

## D-013 — Peek-режим refresh в Server Components
**Контекст.** Next.js запрещает запись cookie в рендере Server Component; ротация
refresh при истёкшем access ломала SSR-страницы (и после неудачной записи браузер
оставался со старым refresh — следующий запрос выглядел как кража токена и
отзывал все сессии).
**Решение.** В `resolveSessionUserId` cookie пишутся до изменения БД; если
запись бросает (read-only контекст RSC) — пользователь опознаётся по действующему
refresh без ротации (peek), а полноценная one-time ротация происходит в ближайшем
Route Handler. Порядок «cookies → БД» гарантирует, что незаписанные токены не
попадают в БД.
**Последствия.** SSR-страницы не пишут cookie; безопасность ротации сохранена
(reuse-детекция без изменений); при редком сбое БД после записи cookie пользователь
просто переавторизуется, каскадного отзыва нет.
